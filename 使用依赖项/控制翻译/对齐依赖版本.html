无标题 # 对齐依赖版本 内容 * [处理不一致的模块版本](#handling_inconsistent_module_versions) * [将本机版本与Gradle对齐](#aligning_versions_natively_with_gradle) * [对齐未随Gradle发布的模块版本](#sec:align-versions-unpublished) 依赖关系版本对齐允许依赖于同一逻辑组（_平台_）的不同模块在依赖关系图中具有相同的版本。 ## [](#handling_inconsistent_module_versions)[处理不一致的模块版本](#handling_inconsistent_module_versions) Gradle支持对齐属于同一“平台”的模块的版本。例如，通常最好是组件的API和实现模块使用相同的版本。但是，由于传递依赖解决方案的博弈，属于同一平台的不同模块最终可能使用不同的版本。例如，您的项目可能取决于`jackson-databind`和`vert.x`库，如下所示： 例子1.声明依赖 `Groovy``Kotlin` build.gradle dependencies \{ // a dependency on Jackson Databind implementation 'com.fasterxml.jackson.core:jackson-databind:2.8.9' // and a dependency on vert.x implementation 'io.vertx:vertx-core:3.5.3' \}   1 dependencies \{ 2 // a dependency on Jackson Databind 3 implementation 'com.fasterxml.jackson.core:jackson-databind:2.8.9' 4 5 // and a dependency on vert.x 6 implementation 'io.vertx:vertx-core:3.5.3' 7 \} 因为`vert.x`取决于`jackson-core`，我们实际上将解决以下依赖项版本： * `jackson-core`版本`2.9.5`（由带来`vertx-core`） * `jackson-databind`版本`2.9.5`（通过冲突解决） * `jackson-annotation`版本`2.9.0`（对的依赖`jackson-databind:2.9.5`） 最终会出现一组不能很好地协同工作的版本，这很容易。为了解决这个问题，Gradle支持依赖关系版本对齐，平台概念支持这种依赖关系对齐。平台代表了一组“一起工作”的模块。或者是因为它们实际上是作为一个整体发布的（当平台的一个成员发布时，所有其他模块也都以相同的版本发布），或者是因为有人测试了这些模块并指出它们可以很好地协同工作（通常，春季平台）。 ## [](#aligning_versions_natively_with_gradle)[将本机版本与Gradle对齐](#aligning_versions_natively_with_gradle) Gradle本机支持Gradle生产的模块的对齐。这是[依赖性约束]()的可传递[性]()的直接结果。因此，如果您具有多项目构建，并且希望使用者获得所有模块的相同版本，那么Gradle提供了一种使用[Java Platform Plugin]()进行此操作的简单方法。 例如，如果您的项目包含3个模块： * `lib` * `utils` * `core`，取决于`lib`和`utils` 并声明以下依赖项的使用者： * `core`版本1.0 * `lib`版本1.1 然后默认情况下，解析将选择`core:1.0`和`lib:1.1`，因为`lib`与无关`core`。我们可以通过在项目中添加一个新的模块_platform_来解决此问题，该模块将对项目的所有模块添加约束： 例子2.平台模块 `Groovy``Kotlin` build.gradle plugins \{ id 'java-platform' \} dependencies \{ // The platform declares constraints on all components that // require alignment constraints \{ api\(project\(":core"\)\) api\(project\(":lib"\)\) api\(project\(":utils"\)\) \} \}   1 plugins \{ 2 id 'java-platform' 3 \} 4 5 dependencies \{ 6 // The platform declares constraints on all components that 7 // require alignment 8 constraints \{ 9 api\(project\(":core"\)\) 10 api\(project\(":lib"\)\) 11 api\(project\(":utils"\)\) 12 \} 13 \} 完成此操作后，我们需要确保所有模块现在都_依赖于该平台_，如下所示： 例子3.声明对平台的依赖 `Groovy``Kotlin` build.gradle dependencies \{ // Each project has a dependency on the platform api\(platform\(project\(":platform"\)\)\) // And any additional dependency required implementation\(project\(":lib"\)\) implementation\(project\(":utils"\)\) \}   1 dependencies \{ 2 // Each project has a dependency on the platform 3 api\(platform\(project\(":platform"\)\)\) 4 5 // And any additional dependency required 6 implementation\(project\(":lib"\)\) 7 implementation\(project\(":utils"\)\) 8 \} 平台对所有组件都包含约束很重要，而且每个组件都对平台具有依赖性是很重要的。这样，每当摇篮将增加一个依赖关系图上的平台的模块，它会_还_包括对平台的其他模块的约束。这意味着，如果看到属于同一平台的另一个模块，我们将自动升级到相同版本。 在我们的示例中，这意味着我们首先看到`core:1.0`，它带来了一个`1.0`对`lib:1.0`and约束的平台`lib:1.0`。然后，我们添加`lib:1.1`对的依赖`platform:1.1`。通过冲突解决，我们选择对`1.1`施加约束的平台`core:1.1`。然后我们在`core:1.0`和之间解决冲突`core:1.1`，这意味着`core`和`lib`现在正确对齐。
仅当您使用Gradle Module元数据时，才对发布的组件强制执行此行为。
## [](#sec:align-versions-unpublished)[对齐未随Gradle发布的模块版本](#sec:align-versions-unpublished) 每当发布者不使用Gradle时（例如在我们的Jackson例子中），我们都可以向Gradle解释说，所有Jackson模块都“属于”同一平台，并且受益于与本机对齐相同的行为。有两个选项可以表示一组模块属于某个平台： 1. 平台作为[BOM]()**发布**，可以使用：例如，可以用作平台。在这种情况下，Gradle缺少的信息是，如果使用平台的成员之一，则应将平台添加到依赖项中。`com.fasterxml.jackson:jackson-bom` 2. 无法使用现有平台。相反，应该由Gradle创建一个**虚拟平台**：在这种情况下，Gradle会根据使用的所有成员来构建平台本身。 要将缺少的信息提供给Gradle，您可以定义[组件元数据规则]()，如下所述。 ### [](#sec:align_bom_platform)[使用已发布的BOM对齐模块的版本](#sec:align_bom_platform) 例子4.依赖版本对齐规则 `Groovy``Kotlin` build.gradle class JacksonBomAlignmentRule implements ComponentMetadataRule \{ void execute\(ComponentMetadataContext ctx\) \{ ctx.details.with \{ if \(id.group.startsWith\("com.fasterxml.jackson"\)\) \{ // declare that Jackson modules belong to the platform defined by the Jackson BOM belongsTo\("com.fasterxml.jackson:jackson-bom:\$\{id.version\}", false\) \} \} \} \}   1 class JacksonBomAlignmentRule implements ComponentMetadataRule \{ 2 void execute\(ComponentMetadataContext ctx\) \{ 3 ctx.details.with \{ 4 if \(id.group.startsWith\("com.fasterxml.jackson"\)\) \{ 5 // declare that Jackson modules belong to the platform defined by the Jackson BOM 6 belongsTo\("com.fasterxml.jackson:jackson-bom:\$\{id.version\}", false\) 7 \} 8 \} 9 \} 10 \} 通过使用`belongsTo`with`false`（**不是**虚拟的），我们声明所有模块都属于同一_发布平台_。在这种情况下，平台是`com.fasterxml.jackson:jackson-bom`，Gradle会像声明其他任何模块一样在声明的存储库中寻找它。 例子5.使用依赖版本对齐规则 `Groovy``Kotlin` build.gradle dependencies \{ components.all\(JacksonBomAlignmentRule\) \}   1 dependencies \{ 2 components.all\(JacksonBomAlignmentRule\) 3 \} 使用该规则，以上示例中的版本将与选定的`com.fasterxml.jackson:jackson-bom`定义版本保持一致。在这种情况下，`com.fasterxml.jackson:jackson-bom:2.9.5`将被选为所选`2.9.5`模块的最高版本。在BOM，下面的版本中定义和将被使用：`jackson-core:2.9.5`，`jackson-databind:2.9.5`和`jackson-annotation:2.9.0`。`jackson-annotation`此处的较低版本可能是所需的结果，因为它是BOM的建议。
从Gradle 6.1开始，此行为工作可靠。实际上，它类似于组件元数据规则，该规则使用来向平台的所有成员添加平台依赖性withDependencies。
### [](#sec:virtual_platform)[在没有发布平台的情况下对齐模块的版本](#sec:virtual_platform) 例子6.依赖版本对齐规则 `Groovy``Kotlin` build.gradle class JacksonAlignmentRule implements ComponentMetadataRule \{ void execute\(ComponentMetadataContext ctx\) \{ ctx.details.with \{ if \(id.group.startsWith\("com.fasterxml.jackson"\)\) \{ // declare that Jackson modules all belong to the Jackson virtual platform belongsTo\("com.fasterxml.jackson:jackson-virtual-platform:\$\{id.version\}"\) \} \} \} \}   1 class JacksonAlignmentRule implements ComponentMetadataRule \{ 2 void execute\(ComponentMetadataContext ctx\) \{ 3 ctx.details.with \{ 4 if \(id.group.startsWith\("com.fasterxml.jackson"\)\) \{ 5 // declare that Jackson modules all belong to the Jackson virtual platform 6 belongsTo\("com.fasterxml.jackson:jackson-virtual-platform:\$\{id.version\}"\) 7 \} 8 \} 9 \} 10 \} 通过使用`belongsTo`不带进一步参数的关键字（平台**是**虚拟的），我们声明所有模块都属于同一_虚拟平台_，该引擎特别对其进行了处理。不会从存储库中检索虚拟平台。在这种情况下`com.fasterxml.jackson:jackson-virtual-platform`，标识符是构建作者自己定义的东西。然后，Gradle通过收集`belongsTo`指向同一虚拟平台的所有语句来动态创建平台的“内容”。 例子7.利用依赖版本对齐规则 `Groovy``Kotlin` build.gradle dependencies \{ components.all\(JacksonAlignmentRule\) \}   1 dependencies \{ 2 components.all\(JacksonAlignmentRule\) 3 \} 使用该规则，以上示例中的所有版本将与对齐`2.9.5`。在这种情况下，也`jackson-annotation:2.9.5`将采用，因为这就是我们定义本地虚拟平台的方式。 对于已发布平台和虚拟平台，Gradle允许您通过在平台上指定_强制_依赖项来覆盖平台本身的版本选择： 例子8.强制平台降级 `Groovy``Kotlin` build.gradle dependencies \{ // Forcefully downgrade the virtual Jackson platform to 2.8.9 implementation enforcedPlatform\('com.fasterxml.jackson:jackson-virtual-platform:2.8.9'\) \}   1 dependencies \{ 2 // Forcefully downgrade the virtual Jackson platform to 2.8.9 3 implementation enforcedPlatform\('com.fasterxml.jackson:jackson-virtual-platform:2.8.9'\) 4 \}
