## [](#version_ordering)[版本订购](#version_ordering) 版本具有隐式排序。版本排序用于： * 确定范围内是否包括特定版本。 * 解决冲突时，确定哪个版本是“最新的”。 版本根据以下规则排序： * 每个版本均分为其组成的“部分”： * 字符`[. - _ +]`用于分隔版本的不同“部分”。 * 同时包含数字和字母的任何部分都将分为以下各个部分：`1a1 == 1.a.1` * 仅比较版本的各个部分。实际的分隔符并不重要：`1.a.1 == 1-a+1 == 1.a-1 == 1a1` * 使用以下规则比较2个版本的等效部分： * 如果两个部分都是数字，则最高数字值**较高**：`1.1`\<`1.2` * 如果一个部分是数字部分，则认为它**高于**非数字部分：`1.a`\<`1.1` * 如果两者都没有数字，这些部分进行比较**的字母顺序，区分大小写**：`1.A`\<`1.B`\<`1.a`\<`1.b` * 具有额外数字部分的版本被认为**高于**没有以下内容的版本：`1.1`\<`1.1.0` * 具有额外非数字部分的版本被认为**低于**没有以下内容的版本：`1.1.a`\<`1.1` * 某些字符串值出于排序目的具有特殊含义： * 字符串`dev`就是考虑**下**比任何其它组部件：`1.0-dev`\<`1.0-alpha`\<`1.0-rc`。 * 琴弦`rc`，`release`和`final`被认为是**较高的**比任何其它组部件（按该顺序排列）：`1.0-zeta`\<`1.0-rc`\<`1.0-release`\<`1.0-final`\<`1.0`。 * 该字符串`SNAPSHOT`有**没有特殊的意义**，并且按字母顺序排序像任何其他字符串的一部分：`1.0-alpha`\<`1.0-SNAPSHOT`\<`1.0-zeta`\<`1.0-rc`\<`1.0`。 * 数字快照版本有**没有特殊的含义**，并进行排序像任何其他数字部分：`1.0`\<`1.0-20150201.121010-123`\<`1.1`。

Gradle 6.5支持替代的，选择加入的版本排序方案，在特殊情况下，它们具有更多后缀：

  • 该字符串SNAPSHOT将被责令高于rc：1.0-RC< 1.0-SNAPSHOT<1.0
  • 该字符串GA会被下一个命令FINAL和RELEASE按字母顺序排列：1.0-RC< 1.0-FINAL< 1.0-GA< 1.0-RELEASE<1.0
  • 字符串SP将责令高于RELEASE，但它仍然比然而不合格版本下，限制了其使用到版本方案兼用FINAL，GA或者RELEASE：1.0-RELEASE< 1.0-SP1<1.0

激活中的功能预览VERSION_ORDERING_V2会settings.gradle(.kts)启用以下一组更改：

enableFeaturePreview\("VERSION\_ORDERING\_V2"\)   1 enableFeaturePreview\("VERSION\_ORDERING\_V2"\)

这些更改将成为Gradle 7.0中的默认设置。

## [](#simple_version_declaration_semantics)[简单的版本声明语义](#simple_version_declaration_semantics) 使用缩写符号声明版本时，例如： 例子1.一个简单的声明 `Groovy``Kotlin` build.gradle dependencies \{ implementation\('org.slf4j:slf4j-api:1.7.15'\) \}   1 dependencies \{ 2 implementation\('org.slf4j:slf4j-api:1.7.15'\) 3 \} 然后，该版本被视为[必需版本]()，这意味着它应为_最低_要求，`1.7.15`但可以由引擎进行升级（乐观升级）。 但是，对于[严格版本]()，有一个简写的表示法，使用的`!!`表示法是： 例子2.严格依赖的简写 `Groovy``Kotlin` build.gradle dependencies \{ // short-hand notation with \!\! implementation\('org.slf4j:slf4j-api:1.7.15\!\!'\) // is equivalent to implementation\("org.slf4j:slf4j-api"\) \{ version \{ strictly '1.7.15' \} \} // or... implementation\('org.slf4j:slf4j-api:\[1.7, 1.8\[\!\!1.7.25'\) // is equivalent to implementation\('org.slf4j:slf4j-api'\) \{ version \{ strictly '\[1.7, 1.8\[' prefer '1.7.25' \} \} \}   1 dependencies \{ 2 // short-hand notation with \!\! 3 implementation\('org.slf4j:slf4j-api:1.7.15\!\!'\) 4 // is equivalent to 5 implementation\("org.slf4j:slf4j-api"\) \{ 6 version \{ 7 strictly '1.7.15' 8 \} 9 \} 10 11 // or... 12 implementation\('org.slf4j:slf4j-api:\[1.7, 1.8\[\!\!1.7.25'\) 13 // is equivalent to 14 implementation\('org.slf4j:slf4j-api'\) \{ 15 version \{ 16 strictly '\[1.7, 1.8\[' 17 prefer '1.7.25' 18 \} 19 \} 20 \} 严格版本_无法升级，_并且会覆盖源自此依赖项提供的任何传递性依赖项。建议对严格版本使用范围。 `[1.7, 1.8[!!1.7.25`上面的表示法等效于： * 严格地`[1.7, 1.8[` * 偏爱`1.7.25` 这意味着引擎**必须**在1.7（包括）和1.8（排除）之间选择一个版本，并且如果图中的其他任何组件都不需要其他版本，则它应该_首选_`1.7.25`。 ## [](#sec:declaring_without_version)[声明没有版本的依赖](#sec:declaring_without_version) 对于较大的项目，建议的做法是声明没有版本的依赖项，并将[依赖项约束]()用于版本声明。优点是，依赖关系约束使您可以在一处管理所有依赖关系的版本，包括传递性。 例子3.声明一个没有版本的依赖 `Groovy``Kotlin` build.gradle dependencies \{ implementation 'org.springframework:spring-web' \} dependencies \{ constraints \{ implementation 'org.springframework:spring-web:5.0.2.RELEASE' \} \}   1 dependencies \{ 2 implementation 'org.springframework:spring-web' 3 \} 4 5 dependencies \{ 6 constraints \{ 7 implementation 'org.springframework:spring-web:5.0.2.RELEASE' 8 \} 9 \}
