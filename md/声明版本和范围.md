# 声明版本和范围

version 6.7.1

内容

  * [版本订购](file:///Users/dxs/temp/gradle-6.7.1/docs/userguide/single_versions.html#version_ordering)
  * [简单的版本声明语义](file:///Users/dxs/temp/gradle-6.7.1/docs/userguide/single_versions.html#simple_version_declaration_semantics)
  * [声明没有版本的依赖](file:///Users/dxs/temp/gradle-6.7.1/docs/userguide/single_versions.html#sec:declaring_without_version)

最简单的版本声明是表示要使用的版本的 _简单字符串_ 。Gradle支持不同的方式来声明版本字符串：

  * 一个确切的版本：比如`1.3`，`1.3.0-beta3`，`1.0-20150201.131010-1`

  * 一个Maven风格的版本范围：例如`[1.0,)`，`[1.1, 2.0)`，`(1.2, 1.5]`

    * 的`[`和`]`符号表示结合的包括端点; `(`并`)`指出排他性界限。

    * 缺少上限或下限时，范围没有上限或下限。

    * 符号`]`可以被用来代替`(`用于排他性下界，并且`[`代替的`)`专用上限。例如`]1.0, 2.0[`

  * 一个 _前缀_ 版本范围：例如`1.+`，`1.3.+`

    * 仅包含与之前部分完全匹配的版本`+`。

    * `+`本身的范围将包括任何版本。

  * 一个`latest-status`版本：例如`latest.integration`，`latest.release`

    * 将使版本最高的模块与指定状态匹配。请参阅[ComponentMetadata.getStatus（）](file:///Users/dxs/temp/gradle-6.7.1/docs/javadoc/org/gradle/api/artifacts/ComponentMetadata.html#getStatus--)。

  * Maven`SNAPSHOT`版本标识符：例如`1.0-SNAPSHOT`，`1.4.9-beta1-SNAPSHOT`

__ |

Gradle 6.5支持版本范围的备用选择加入行为。

当上限排除版本时，它还充当前缀排除。这意味着`[1.0,
2.0[`还将排除所有所有`2.0`小于的版本`2.0`。例如，例如`2.0-dev1`或`2.0-SNAPSHOT`不再包含在范围内。

激活预览功能`VERSION_ORDERING_V2`在`settings.gradle(.kts)`使这种变化：

    
    
    enableFeaturePreview("VERSION_ORDERING_V2")

此更改将成为Gradle 7.0中的默认设置。

##
[](file:///Users/dxs/temp/gradle-6.7.1/docs/userguide/single_versions.html#version_ordering)[版本订购](file:///Users/dxs/temp/gradle-6.7.1/docs/userguide/single_versions.html#version_ordering)

版本具有隐式排序。版本排序用于：

  * 确定范围内是否包括特定版本。

  * 解决冲突时，确定哪个版本是“最新的”。

版本根据以下规则排序：

  * 每个版本均分为其组成的“部分”：

    * 字符`[. - _ +]`用于分隔版本的不同“部分”。

    * 同时包含数字和字母的任何部分都将分为以下各个部分： `1a1 == 1.a.1`

    * 仅比较版本的各个部分。实际的分隔符并不重要：`1.a.1 == 1-a+1 == 1.a-1 == 1a1`

  * 使用以下规则比较2个版本的等效部分：

    * 如果两个部分都是数字，则最高数字值 **较高** ：`1.1`<`1.2`

    * 如果一个部分是数字部分，则认为它 **高于** 非数字部分：`1.a`<`1.1`

    * 如果两者都没有数字，这些部分进行比较 **的字母顺序，区分大小写** ：`1.A`< `1.B`< `1.a`<`1.b`

    * 具有多余数字部分的版本被认为 **高于** 没有以下内容的版本：`1.1`<`1.1.0`

    * 具有额外非数字部分的版本被认为 **低于** 没有以下内容的版本：`1.1.a`<`1.1`

  * 某些字符串值出于排序目的具有特殊含义：

    * 字符串`dev`就是考虑 **下** 比任何其它组部件：`1.0-dev`< `1.0-alpha`< `1.0-rc`。

    * The strings `rc`, `release` and `final` are considered **higher** than any other string part (sorted in that order): `1.0-zeta` < `1.0-rc` < `1.0-release` < `1.0-final` < `1.0`.

    * The string `SNAPSHOT` has **no special meaning** , and is sorted alphabetically like any other string part: `1.0-alpha` < `1.0-SNAPSHOT` < `1.0-zeta` < `1.0-rc` < `1.0`.

    * Numeric snapshot versions have **no special meaning** , and are sorted like any other numeric part: `1.0` < `1.0-20150201.121010-123` < `1.1`.

__ |

Gradle 6.5 supports an alternate, opt-in, version ordering scheme which
special cases more suffixes:

  * The string `SNAPSHOT` will be ordered higher than `rc`: `1.0-RC` < `1.0-SNAPSHOT` < `1.0`

  * 该字符串`GA`会被下一个命令`FINAL`和`RELEASE`按字母顺序排列：`1.0-RC`< `1.0-FINAL`< `1.0-GA`< `1.0-RELEASE`<`1.0`

  * 字符串`SP`将责令高于`RELEASE`，但它仍然比然而不合格版本下，限制了其使用到版本方案兼用`FINAL`，`GA`或者`RELEASE`：`1.0-RELEASE`< `1.0-SP1`<`1.0`

激活中的功能预览`VERSION_ORDERING_V2`会`settings.gradle(.kts)`启用以下一组更改：

    
    
    enableFeaturePreview("VERSION_ORDERING_V2")

这些更改将成为Gradle 7.0中的默认设置。

    }

然后，该版本被视为[必需版本](file:///Users/dxs/temp/gradle-6.7.1/docs/userguide/rich_versions.html#sec:required-
version)，这意味着它应为 _最低_ 要求，`1.7.15`但可以由引擎进行升级（乐观升级）。

但是，对于[严格版本](file:///Users/dxs/temp/gradle-6.7.1/docs/userguide/rich_versions.html#sec:strict-
version)，有一个简写的表示法，使用的`!!`表示法是：

例子2.严格依赖的简写

`Groovy``Kotlin`

build.gradle

    
    
    dependencies {
        // short-hand notation with !!
        implementation('org.slf4j:slf4j-api:1.7.15!!')
        // is equivalent to
        implementation("org.slf4j:slf4j-api") {
            version {
               strictly '1.7.15'
            }
        }
    
        // or...
        implementation('org.slf4j:slf4j-api:[1.7, 1.8[!!1.7.25')
        // is equivalent to
        implementation('org.slf4j:slf4j-api') {
            version {
               strictly '[1.7, 1.8['
               prefer '1.7.25'
            }
        }
    }

build.gradle.kts

    
    
    dependencies {
        // short-hand notation with !!
        implementation("org.slf4j:slf4j-api:1.7.15!!")
        // is equivalent to
        implementation("org.slf4j:slf4j-api") {
            version {
               strictly("1.7.15")
            }
        }
    
        // or...
        implementation("org.slf4j:slf4j-api:[1.7, 1.8[!!1.7.25")
        // is equivalent to
        implementation("org.slf4j:slf4j-api") {
            version {
               strictly("[1.7, 1.8[")
               prefer("1.7.25")
            }
        }
    }

严格版本 _无法升级，_ 并且会覆盖源自此依赖项提供的任何传递性依赖项。建议对严格版本使用范围。

`[1.7, 1.8[!!1.7.25`上面的表示法等效于：

  * 严格地 `[1.7, 1.8[`

  * 偏爱 `1.7.25`

这意味着引擎 **必须** 在1.7（包括）和1.8（排除）之间选择一个版本，并且如果图中的其他任何组件都不需要其他版本，则它应该 _首选_
`1.7.25`。

##
[](file:///Users/dxs/temp/gradle-6.7.1/docs/userguide/single_versions.html#sec:declaring_without_version)[声明没有版本的依赖](file:///Users/dxs/temp/gradle-6.7.1/docs/userguide/single_versions.html#sec:declaring_without_version)

对于较大的项目，建议的做法是声明没有版本的依赖项，并将[依赖项约束](file:///Users/dxs/temp/gradle-6.7.1/docs/userguide/dependency_constraints.html#sec:adding-
constraints-transitive-deps)用于版本声明。优势在于，依赖关系约束使您可以在一处管理所有依赖关系的版本，包括可传递的依赖关系。

例子3.声明一个没有版本的依赖

`Groovy``Kotlin`

build.gradle

    
    
    dependencies {
        implementation 'org.springframework:spring-web'
    }
    
    dependencies {
        constraints {
            implementation 'org.springframework:spring-web:5.0.2.RELEASE'
        }
    }

build.gradle.kts

    
    
    dependencies {
        implementation("org.springframework:spring-web")
    }
    
    dependencies {
        constraints {
            implementation("org.springframework:spring-web:5.0.2.RELEASE")
        }
    }
