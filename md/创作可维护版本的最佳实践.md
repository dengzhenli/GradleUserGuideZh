Gradle具有丰富的API，其中包含创建构建逻辑的几种方法。关联的灵活性很容易导致不必要的复杂的构建，而通常直接将自定义代码添加到构建脚本中。在本章中，我们介绍了几种最佳实践，它们将帮助您开发易于使用的表达性和可维护性版本。

## [](#sec:avoid_imperative_logic_in_scripts)[避免在脚本中使用命令式逻辑](#sec:avoid_imperative_logic_in_scripts)

Gradle运行时不对构建逻辑强制执行特定样式。因此，很容易以将声明性DSL元素与命令性程序代码混合在一起的构建脚本结束。让我们谈论一些具体的例子。

  * _声明性代码：_ 内置的，与语言无关的DSL元素（例如[Project.dependencies {}](https://docs.gradle.org/6.7.1/dsl/org.gradle.api.Project.html#org.gradle.api.Project:dependencies\(groovy.lang.Closure\))或[Project.repositories {}](https://docs.gradle.org/6.7.1/dsl/org.gradle.api.Project.html#org.gradle.api.Project:repositories\(groovy.lang.Closure\))）或插件公开的DSL

  * _命令式代码：_ 条件逻辑或非常复杂的任务动作实现

每个构建脚本的最终目标应该是仅包含声明性语言元素，这使代码更易于理解和维护。祈使逻辑应该存在于二进制插件中，然后将其应用于构建脚本。作为辅助产品，如果您将工件发布到二进制存储库，则可以自动使您的团队[在其他项目中重用插件逻辑](https://docs.gradle.org/6.7.1/samples/sample_publishing_convention_plugins.html)。

以下示例构建显示了直接在构建脚本中使用条件逻辑的否定示例。尽管此代码段很小，但是很容易想象使用大量过程语句的成熟的构建脚本及其对可读性和可维护性的影响。通过将代码移到一个类中，也可以单独对其进行测试。

例子1.使用条件逻辑创建任务的构建脚本

`Groovy``Kotlin`

build.gradle

    
    
    if (project.findProperty('releaseEngineer') != null) {
        tasks.register('release') {
            doLast {
                logger.quiet 'Releasing to production...'
    
                // release the artifact to production
            }
        }
    }

build.gradle.kts

    
    
    if (project.findProperty("releaseEngineer") != null) {
        tasks.register("release") {
            doLast {
                logger.quiet("Releasing to production...")
    
                // release the artifact to production
            }
        }
    }

让我们将构建脚本与实现为二进制插件的相同逻辑进行比较。该代码起初看起来可能涉及更多，但显然看起来更像典型的应用程序代码。这个特定的插件类位于[`buildSrc`目录中](https://docs.gradle.org/6.7.1/userguide/organizing_gradle_projects.html#sec:build_sources)，该[目录](https://docs.gradle.org/6.7.1/userguide/organizing_gradle_projects.html#sec:build_sources)可自动用于构建脚本。

例子2.一个实现命令式逻辑的二进制插件

ReleasePlugin.java

    
    
    package com.enterprise;
    
    import org.gradle.api.Action;
    import org.gradle.api.Plugin;
    import org.gradle.api.Project;
    import org.gradle.api.Task;
    import org.gradle.api.tasks.TaskProvider;
    
    public class ReleasePlugin implements Plugin<Project> {
        private static final String RELEASE_ENG_ROLE_PROP = "releaseEngineer";
        private static final String RELEASE_TASK_NAME = "release";
    
        @Override
        public void apply(Project project) {
            if (project.findProperty(RELEASE_ENG_ROLE_PROP) != null) {
                Task task = project.getTasks().create(RELEASE_TASK_NAME);
    
                task.doLast(new Action<Task>() {
                    @Override
                    public void execute(Task task) {
                        task.getLogger().quiet("Releasing to production...");
    
                        // release the artifact to production
                    }
                });
            }
        }
    }

现在，构建逻辑已转换为插件，您可以在构建脚本中应用它。该构建脚本已从8行代码缩减为一个内衬。

例子3.一个构建脚本，它使用一个封装命令式逻辑的插件

`Groovy``Kotlin`

build.gradle

    
    
    plugins {
        id 'com.enterprise.release'
    }

build.gradle.kts

    
    
    plugins {
        id("com.enterprise.release")
    }

### [](#alternatives_for_oft_used_internal_apis)[常用内部API的替代方案](#alternatives_for_oft_used_internal_apis)

要为您的自定义任务提供嵌套的DSL，请不要使用`org.gradle.internal.reflect.Instantiator`;
改用[ObjectFactory](https://docs.gradle.org/6.7.1/javadoc/org/gradle/api/model/ObjectFactory.html)。阅读[有关惰性配置的章节](https://docs.gradle.org/6.7.1/userguide/lazy_configuration.html#lazy_configuration)可能也会有所帮助。

不要使用`org.gradle.api.internal.ConventionMapping`。使用[提供者](https://docs.gradle.org/6.7.1/javadoc/org/gradle/api/provider/Provider.html)和/或[财产](https://docs.gradle.org/6.7.1/javadoc/org/gradle/api/provider/Property.html)。您可以在[实现插件指南中](https://guides.gradle.org/implementing-gradle-plugins/#capturing_user_input_to_configure_plugin_runtime_behavior)找到捕获用户输入以配置运行时行为的示例。

而是`org.gradle.internal.os.OperatingSystem`使用另一种方法来检测操作系统，例如[Apache commons-lang SystemUtils](https://commons.apache.org/proper/commons-lang/apidocs/org/apache/commons/lang3/SystemUtils.html)或`System.getProperty("os.name")`。

使用其他集合或I /O架构，而不是`org.gradle.util.CollectionUtils`，`org.gradle.util.GFileUtils`和其他类下`org.gradle.util.*`。

Gradle插件作者可能会发现“设计Gradle插件”小节对[将插件实现限制为Gradle的公共API](https://guides.gradle.org/designing-gradle-plugins/#restricting_the_plugin_implementation_to_gradle_s_public_api)有所帮助。

## [](#sec:declaring_tasks)[声明任务时遵循约定](#sec:declaring_tasks)

任务API为构建作者提供了很大的灵活性，使其可以在构建脚本中声明任务。为了获得最佳的可读性和可维护性，请遵循以下规则：

  * 任务类型应该是任务名称后面括号内的唯一键值对。

  * 其他配置应在任务的配置块中完成。

  * [](https://docs.gradle.org/6.7.1/userguide/tutorial_using_tasks.html#sec:hello_world)声明任务时添加的[任务动作](https://docs.gradle.org/6.7.1/userguide/tutorial_using_tasks.html#sec:hello_world)只能使用[Task.doFirst {}](https://docs.gradle.org/6.7.1/dsl/org.gradle.api.Task.html#org.gradle.api.Task:doFirst\(org.gradle.api.Action\))或[Task.doLast {}](https://docs.gradle.org/6.7.1/dsl/org.gradle.api.Task.html#org.gradle.api.Task:doLast\(org.gradle.api.Action\))方法声明。

  * 在声明临时任务（没有显式类型的任务）时，如果仅声明单个操作，则应使用[Task.doLast {}](https://docs.gradle.org/6.7.1/dsl/org.gradle.api.Task.html#org.gradle.api.Task:doLast\(org.gradle.api.Action\))。

  * 一个任务应该[定义一个组和描述](#sec:improving_task_discoverability)。

例子4.遵循最佳实践的任务定义

`Groovy``Kotlin`

build.gradle

    
    
    import com.enterprise.DocsGenerate
    
    def generateHtmlDocs = tasks.register('generateHtmlDocs', DocsGenerate) {
        group = JavaBasePlugin.DOCUMENTATION_GROUP
        description = 'Generates the HTML documentation for this project.'
        title = 'Project docs'
        outputDir = file("$buildDir/docs")
    }
    
    tasks.register('allDocs') {
        group = JavaBasePlugin.DOCUMENTATION_GROUP
        description = 'Generates all documentation for this project.'
        dependsOn generateHtmlDocs
    
        doLast {
            logger.quiet('Generating all documentation...')
        }
    }

build.gradle.kts

    
    
    import com.enterprise.DocsGenerate
    
    tasks.register<DocsGenerate>("generateHtmlDocs") {
        group = JavaBasePlugin.DOCUMENTATION_GROUP
        description = "Generates the HTML documentation for this project."
        title = "Project docs"
        outputDir = file("$buildDir/docs")
    }
    
    tasks.register("allDocs") {
        group = JavaBasePlugin.DOCUMENTATION_GROUP
        description = "Generates all documentation for this project."
        dependsOn("generateHtmlDocs")
    
        doLast {
            logger.quiet("Generating all documentation...")
        }
    }

## [](#sec:improving_task_discoverability)[提高任务发现能力](#sec:improving_task_discoverability)

即使是构建的新用户，也应该能够快速而轻松地找到关键信息。在Gradle中，您可以为构建的任何任务声明一个[组](https://docs.gradle.org/6.7.1/dsl/org.gradle.api.Project.html#org.gradle.api.Project:group)和一个[描述](https://docs.gradle.org/6.7.1/dsl/org.gradle.api.Project.html#org.gradle.api.Project:description)。在[任务报告](https://docs.gradle.org/6.7.1/userguide/command_line_interface.html#sec:listing_tasks)使用指定的值来组织和呈现，便于发现性的任务。分配组和描述对于您希望构建用户调用的任何任务最有帮助。

示例任务`generateDocs`以HTML页面的形式为项目生成文档。该任务应组织在水桶下方`Documentation`。该描述应表达其意图。

例子5.声明组和描述的任务

`Groovy``Kotlin`

build.gradle

    
    
    tasks.register('generateDocs') {
        group = 'Documentation'
        description = 'Generates the HTML documentation for this project.'
    
        doLast {
            // action implementation
        }
    }

build.gradle.kts

    
    
    tasks.register("generateDocs") {
        group = "Documentation"
        description = "Generates the HTML documentation for this project."
    
        doLast {
            // action implementation
        }
    }

任务报告的输出反映了分配的值。

    
    
    > gradle tasks
    
    > Task :tasks
    
    Documentation tasks
    -------------------
    generateDocs - Generates the HTML documentation for this project.

## [](#sec:minimize_logic_executed_configuration_phase)[最小化在配置阶段执行的逻辑](#sec:minimize_logic_executed_configuration_phase)

对于每个构建脚本开发人员而言，了解[构建生命周期](https://docs.gradle.org/6.7.1/userguide/build_lifecycle.html#build_lifecycle)的不同阶段及其对构建逻辑的性能和评估顺序的影响都是至关重要的。在配置阶段，应
_配置_ 项目及其域对象，而执行阶段仅执行命令行中请求的任务的动作及其依赖项。请注意，任何不属于任务动作的代码都将在 _每次运行_
构建时执行。一个[构建扫描](https://scans.gradle.com/get-
started)可帮助您识别每个生命周期阶段的过程中花费的时间。这是诊断常见性能问题的宝贵工具。

让我们考虑下面对上述反模式的描述。在构建脚本中，您可以看到分配给配置的依赖关系`printArtifactNames`在任务操作之外已解决。

例子6.应该避免在配置过程中执行逻辑

`Groovy``Kotlin`

build.gradle

    
    
    dependencies {
        implementation 'log4j:log4j:1.2.17'
    }
    
    tasks.register('printArtifactNames') {
        // always executed
        def libraryNames = configurations.compileClasspath.collect { it.name }
    
        doLast {
            logger.quiet libraryNames
        }
    }

build.gradle.kts

    
    
    dependencies {
        implementation("log4j:log4j:1.2.17")
    }
    
    tasks.register("printArtifactNames") {
        // always executed
        val libraryNames = configurations.compileClasspath.get().map { it.name }
    
        doLast {
            logger.quiet(libraryNames.toString())
        }
    }

解决依赖关系的代码应移到任务操作中，以避免在实际需要依赖关系之前解决依赖关系带来的性能影响。

例子7.最好在执行阶段执行逻辑

`Groovy``Kotlin`

build.gradle

    
    
    dependencies {
        implementation 'log4j:log4j:1.2.17'
    }
    
    tasks.register('printArtifactNames') {
        doLast {
            def libraryNames = configurations.compileClasspath.collect { it.name }
            logger.quiet libraryNames
        }
    }

build.gradle.kts

    
    
    dependencies {
        implementation("log4j:log4j:1.2.17")
    }
    
    tasks.register("printArtifactNames") {
        doLast {
            val libraryNames = configurations.compileClasspath.get().map { it.name }
            logger.quiet(libraryNames.toString())
        }
    }

## [](#sec:avoiding_use_of_gradlebuild)[避免使用`GradleBuild`任务类型](#sec:avoiding_use_of_gradlebuild)

该[GradleBuild](https://docs.gradle.org/6.7.1/dsl/org.gradle.api.tasks.GradleBuild.html)任务类型允许构建脚本定义一个任务调用另一个Gradle构建。通常不建议使用此类型。在某些极端情况下，所调用的构建不会像从命令行或通过Tooling
API公开相同的运行时行为，从而导致意外结果。

通常，有一种更好的方法来对需求进行建模。适当的方法取决于当前的问题。这里有一些选择：

  * 如果打算将来自不同模块的任务作为统一构建执行，则将构建建模为[多项目](https://docs.gradle.org/6.7.1/userguide/multi_project_builds.html#multi_project_builds)构建。

  * 对物理上分离但偶尔应作为一个单元[构建](https://docs.gradle.org/6.7.1/userguide/composite_builds.html#composite_builds)的项目使用[复合构建](https://docs.gradle.org/6.7.1/userguide/composite_builds.html#composite_builds)。

## [](#sec:avoiding_inter_project_configuration)[避免项目间配置](#sec:avoiding_inter_project_configuration)

Gradle并不限制构建脚本作者从一个项目进入[多](https://docs.gradle.org/6.7.1/userguide/multi_project_builds.html#multi_project_builds)模型构建中的另一个项目的领域模型。高度耦合的项目会损害[构建执行性能](https://docs.gradle.org/6.7.1/userguide/multi_project_configuration_and_execution.html#sec:parallel_execution)以及代码的可读性和可维护性。

应避免以下做法：

  * 通过[Task.dependsOn（java.lang.Object ...）](https://docs.gradle.org/6.7.1/dsl/org.gradle.api.Task.html#org.gradle.api.Task:dependsOn\(java.lang.Object\[\]\))显式依赖于另一个项目中的任务。

  * 在另一个项目的域对象上设置属性值或调用方法。

  * 用[GradleBuild](#sec:avoiding_use_of_gradlebuild)执行构建的另一部分。

  * 声明不必要的[项目依赖项](https://docs.gradle.org/6.7.1/userguide/declaring_dependencies.html#sub:project_dependencies)。

## [](#sec:avoiding_passwords_in_plain_text)[外部化和加密您的密码](#sec:avoiding_passwords_in_plain_text)

大多数构建需要使用一个或多个密码。此需求的原因可能有所不同。一些构建需要密码才能将工件发布到安全的二进制存储库，而其他构建则需要密码来下载二进制文件。密码应始终保持安全，以防止欺诈。在任何情况下，都不应以纯文本形式将密码添加到构建脚本中，也不要`gradle.properties`在项目目录中的文件中声明该密码。这些文件通常位于版本控制存储库中，任何有权访问该文件的人都可以查看。

密码以及任何其他敏感数据应保存在版本控制的项目文件外部。Gradle公开了一个用于在[ProviderFactory中](https://docs.gradle.org/6.7.1/javadoc/org/gradle/api/provider/ProviderFactory.html#credentials-java.lang.Class-java.lang.String-)提供凭证的API
以及[Artifact存储库](https://docs.gradle.org/6.7.1/userguide/declaring_repositories.html#sec:handling_credentials)
，该API允许
在构建需要它们时使用[Gradle属性](https://docs.gradle.org/6.7.1/userguide/build_environment.html#sec:gradle_configuration_properties)提供凭证值。这样，凭据可以存储在`gradle.properties`用户主目录中的文件中，也可以使用命令行参数或环境变量注入到构建中。

如果您将敏感凭据存储在用户主目录中`gradle.properties`，请考虑对其进行加密。目前，Gradle还没有提供用于加密，存储和访问密码的内置机制。解决此问题的一个很好的解决方案是[Gradle Credentials插件](https://github.com/etiennestuder/gradle-credentials-plugin)。
